## 決定論に関する予防的メモ (M0 設計時点)

- **Go の map 反復はランダム順**。トレース・送信順序・投票集計に map range を
  使うと同一シードでも実行ごとに挙動が変わる。→ DECISIONS.md D-006。
- **encoding/gob / encoding/json の map エンコードも非決定**。→ D-005。
- **スライスのエイリアシング**: ノード間で Entry スライスを共有すると、
  片方の append がもう片方の backing array を破壊しうる。ログへの取り込み時と
  メッセージ構築時は必ずコピーする(Entry.Data は不変として扱い共有可)。
- **heap のタイブレーク**: (time, seq) の複合キーにしないと同時刻イベントの
  順序が heap の内部構造依存になり非決定化する。

## M1: Leader Completeness チェッカーの偽陽性 (論文 §5.4.3 の読み違え)

「現在の全リーダーは全 commit 済みエントリを持つ」と検査したら偽陽性が出た。
分断中でまだ退位していない古い term の stale リーダーは、より新しい term で
commit されたエントリを持たなくて**正当**。論文の性質は「エントリを commit した
term より後の term のリーダー」にのみ要求される。

チェッカー側の対処: commit を観測したノードの currentTerm を「commit した
リーダーの term の健全な上界」として記録し (観測のたびに min で引き締め)、
その上界より高い term のリーダーだけを検査する。教訓: **安全性チェッカー自体が
分散システムの部分観測性を尊重しないと、正しい実装を偽って告発する**。

## M1: リーダーの送信エントリはコピー必須

sendAppend でログの backing array をそのまま Message.Entries に載せると、
リーダーが後で follower に転落して truncate → append した際に、在空メッセージの
中身が書き換わる (Go スライスのエイリアシング)。log.slice() は必ずコピーを返す。
