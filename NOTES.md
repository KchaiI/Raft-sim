# NOTES.md — 実装中の教訓

ハマったバグ、論文の読み違え、シミュレータ設計の落とし穴を随時記録する。

## 決定論に関する予防的メモ (M0 設計時点)

- **Go の map 反復はランダム順**。トレース・送信順序・投票集計に map range を
  使うと同一シードでも実行ごとに挙動が変わる。→ DECISIONS.md D-006。
- **encoding/gob / encoding/json の map エンコードも非決定**。→ D-005。
- **スライスのエイリアシング**: ノード間で Entry スライスを共有すると、
  片方の append がもう片方の backing array を破壊しうる。ログへの取り込み時と
  メッセージ構築時は必ずコピーする(Entry.Data は不変として扱い共有可)。
- **heap のタイブレーク**: (time, seq) の複合キーにしないと同時刻イベントの
  順序が heap の内部構造依存になり非決定化する。

## M1: Leader Completeness チェッカーの偽陽性 (論文 §5.4.3 の読み違え)

「現在の全リーダーは全 commit 済みエントリを持つ」と検査したら偽陽性が出た。
分断中でまだ退位していない古い term の stale リーダーは、より新しい term で
commit されたエントリを持たなくて**正当**。論文の性質は「エントリを commit した
term より後の term のリーダー」にのみ要求される。

チェッカー側の対処: commit を観測したノードの currentTerm を「commit した
リーダーの term の健全な上界」として記録し (観測のたびに min で引き締め)、
その上界より高い term のリーダーだけを検査する。教訓: **安全性チェッカー自体が
分散システムの部分観測性を尊重しないと、正しい実装を偽って告発する**。

## M4: ログエントリのペイロード形式は最初から一本化する

M2 の複製駆動ワークロードは不透明バイト列 (`cmd-N`) を提案していたが、
M4 で apply 層が全 EntryNormal を kv.Command としてデコードするようになり、
旧形式のペイロードで即 panic した。エントリのペイロード形式はマイルストーンを
またいで共有される契約なので、途中から導入するアプリ層のデコーダが読める形式に
最初から揃えておくべきだった (予約 ClientID=0 の Put に変更して解決)。

## M5: InstallSnapshot はチェッカーの増分スキャンも巻き込む

InstallSnapshot でログが丸ごと置き換わると FirstIndex が「前回観測の末尾」を
飛び越えて進む。増分ログ検査の再開位置をクランプしないと圧縮済み領域を
EntryAt で引いて nil → 偽の「ログ穴」違反になる。増分チェッカーは
「観測対象が不連続にジャンプする遷移 (スナップショット・再起動)」を
必ず列挙して個別に扱うこと。

## M5: 拒否応答で pendingSnap を解除しない

スナップショット送信中に届く古い AppendEntries 拒否 (ハートビートへの拒否の
重複配送など) で pendingSnap を解除すると、拒否 1 通ごとにスナップショット
全体を再送してしまう。解除は成功応答 (SnapResp / AppResp success) のみ。
喪失時の再送は CheckQuorum 周期のリセットが担保する。

## M6: TimeoutNow 送信後も transferee を保持する (博士論文 §3.10 の読み違え)

移譲対象が追いついた時点で TimeoutNow を送り、そこで transferee をクリアして
いたら、リーダーがその後もクライアント提案を受け続け、対象のログが常に一歩
遅れて選挙に勝てないことがあった (テストで顕在化: 移譲失敗 → 旧リーダー残留)。
博士論文の記述通り「移譲の完了 (高 term の観測で退位) かタイムアウトまで
新規提案を受けない」を維持する必要がある。

## M6: single-server change の検査はリーダー在任中に限定する

「構成は一度に 1 ノードしか変わらない」を全ノードの観測ごとに検査すると
偽陽性になる: catch-up 中のフォロワーは 1 つの AppendEntries バッチで複数の
構成エントリを受け取り、構成が一気に複数段進む (truncate での巻き戻りもある)。
規律が成り立つのは構成エントリを「生成」するリーダーの連続在任中のみ。

## M6: 手動シナリオテストでは「クラッシュ」を実際に再現する

Figure 8 テストで S1 を「以後メッセージを配らない」だけで済ませたら、
S1 はリーダーのままなので TimeoutNow を無視し、選挙駆動が空振りした。
クラッシュの模擬は RestartNode(永続状態のみ) で作り直すこと。
leader stickiness / votedFor の衝突があるため、手動選挙は複数ラウンド
(term を進めながら) 試す必要があるのも罠。

## M1: リーダーの送信エントリはコピー必須

sendAppend でログの backing array をそのまま Message.Entries に載せると、
リーダーが後で follower に転落して truncate → append した際に、在空メッセージの
中身が書き換わる (Go スライスのエイリアシング)。log.slice() は必ずコピーを返す。
