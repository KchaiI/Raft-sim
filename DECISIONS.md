# DECISIONS.md — 仕様の曖昧点に対する判断記録

仕様 (SPEC.md) に明記されていない点について、ブロックせずに下した判断とその理由。

## D-001: 仮想時間の単位はマイクロ秒 (int64)

SPEC は仮想時間の単位を規定していない。ミリ秒だと指数分布遅延の解像度が粗く
イベント時刻のタイ(同時刻)が多発する。マイクロ秒なら遅延分布を自然に表現でき、
タイはイベントキューの挿入順序 seq で決定論的に解決する。

## D-002: raft パッケージは `math/rand` を一切 import しない

SPEC §6 は「`math/rand` (グローバル)」の禁止と書く。`rand.New(rand.NewSource(seed))`
のためだけの import を許す読み方もできるが、機械検査の単純さと安全側に倒し、
raft パッケージでは `math/rand` の import 自体を禁止する。乱数は
`raft.RNG` インターフェース(`Intn(n int) int`)としてシミュレータから注入する。
禁止 import は `time` / `math/rand` / `math/rand/v2` / `sync` / `os` / `net` 系すべて。

## D-003: fsync 境界クラッシュのモデル化

1 つの `Output.Persist` = 1 fsync 単位(原子的)。fsync 前クラッシュは
「その Persist が耐久ストレージに一切反映されず、かつ同一 Step が生成した
送信メッセージ・apply もすべて消滅する」としてモデル化する。
理由: Raft の正しさは「状態を永続化してから返信する」ことに依存するため、
fsync 完了前に外部作用(メッセージ送信)が観測されてはならない。シミュレータは
Step の出力処理時に p_crash_persist の確率でこの巻き戻しクラッシュを注入する。

## D-004: クロックスキューは「ノードごとの固定倍率 ±20%」

SPEC は「ノードごとのタイマー速度を±20%変動」とのみ書く。実装は各ノードの
tick 間隔に run 開始時(および再起動時)に [0.8, 1.2] の倍率を掛ける。
時間経過による drift の変動は入れない(選出タイムアウトのランダム化と合わせれば
タイマー速度差の検証としては十分)。

## D-005: 決定論的シリアライズ(gob / JSON 不使用)

スナップショットやコマンドのエンコードに encoding/gob を使うと map の
反復順序で出力バイトが非決定になる。自作の長さ接頭辞つきバイナリコーデック
(キーをソートして書き出す)を使う。これはトレースのバイト一致要件 (M0) と
1シード=1宇宙の決定論保証に必須。

## D-006: map の range 禁止(決定論に影響する箇所)

Go の map 反復順序はランダム。投票集計・quorum 判定・ピアへの送信などは
すべてソート済みスライス(config.Voters は昇順維持)を反復する。
シミュレータ・チェッカー内でも、動作順序やトレース出力に影響する箇所では
map の range を使わない(使う場合はキーをソートしてから)。

## D-007: クライアント⇔サーバー通信の障害モデル

SPEC §3.3 の障害モデルはノード間メッセージについて規定する。クライアント要求も
同じネットワーク(喪失/重複/遅延)を通すが、ネットワーク分断の対象には含めない
(クライアントは全ノードに到達可能)。分断されたリーダーがクライアントに応答
できないことは「commit できない → 応答が返らずタイムアウト」で自然に検証される。
クライアントの再送(同一 clientID+seqNo)が重複リクエスト排除の検証を兼ねる。

## D-008: ReadIndex は実装しない

SPEC §2 で任意とされているため。線形化 Read は Get もログを経由させて成立させる。

## D-009: Raft コアは tick 駆動

タイマーは「シミュレータが Tick 入力を届け、ノードが tick 数を数える」方式。
election timeout は [ET, 2·ET) tick でランダム化(RNG 注入)。wall clock への
依存が構造的に存在しない。

## D-010: 新サーバーの catch-up フェーズ

博士論文 §4.2.1 に従い、AddServer は (1) 新サーバーを非投票メンバーとして
ログ複製で追いつかせ、(2) 追いついた時点で構成変更エントリを append する。
「追いついた」の判定は matchIndex がリーダーの lastIndex の近傍
(election timeout 内に複製しきれる程度の残差)に達したこと、とする。

## D-011: スナップショットは 1 メッセージで転送

SPEC はチャンク分割を要求しないため、InstallSnapshot は snapshot 全体を
1 つのメッセージで送る。転送中クラッシュ・喪失はメッセージ喪失 + 再送で
カバーされる(受け入れシナリオ「スナップショット転送中のクラッシュ」は
リーダー/フォロワー双方のクラッシュを狙い撃ちで注入して検証する)。

## D-012: カバレッジの測定方法

「Raft コアの単体テストカバレッジ 90% 以上」は、リポジトリ全テストを
`-coverpkg=./raft` で実行した合算カバレッジで判定する(raft パッケージの
振る舞いの多くは sim 経由の統合テストで最も自然に到達するため)。

## D-013: 分断は送信時点で判定

分断中のノードペア間のメッセージは送信時に落とす。分断発生時点で在空
(in-flight)のメッセージは配送される(遅延した古いメッセージの到着として
現実にも起こる状況であり、むしろ検証として価値がある)。

## D-014: commitIndex は永続化しない

論文 Figure 2 の通り currentTerm / votedFor / log[] のみ永続化。再起動後の
commitIndex はスナップショットの index から復元し、以降は AppendEntries の
leaderCommit で回復する。チェッカーの「commitIndex 単調性」は再起動イベントを
またぐ場合のみ低下を許す。
