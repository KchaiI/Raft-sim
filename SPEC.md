# Raft 実装 + 決定論的障害注入検証 — 仕様書

## 1. ゴール

Raft合意アルゴリズム(Ongaro & Ousterhout, 2014 + 博士論文)を**論文から**Goで完全実装し、
決定論的シミュレーションによる障害注入で「正しさ」を機械的に証明する。

最終的な成功宣言は次の1行で行えること:

> **10,000シード × ランダム障害シナリオを実行し、安全性不変条件・線形化可能性の違反ゼロ。
> 任意の失敗はシード指定で100%再現可能。**

### ゴールでないもの (Non-goals)

- 実ネットワーク・実プロセスでの動作 (シミュレータ内で完結する)
- 性能最適化 (正しさが唯一の評価軸。スループットは測定するが目標値は持たない)
- 外部Raftライブラリの参照・流用 (etcd/raft等のコードは見ない。論文のみを一次資料とする)

## 3. アーキテクチャ

**核となる設計判断: Raftコアは純粋な決定論的ステートマシンとする。**
goroutine・wall clock・グローバル乱数を一切持たず、入力イベント(メッセージ受信、
タイマー発火)を受けて出力(送信すべきメッセージ、永続化すべき状態、applyすべきエントリ)を
返すだけの構造にする。これにより、シミュレータが時間と通信を完全に支配できる。

```
raft/
├── raft/        Raftコア (純粋ステートマシン。I/Oなし・時計なし・乱数は注入)
├── storage/     永続状態の抽象 + シミュレート耐久ストレージ (クラッシュ時にfsync済みデータのみ生存)
├── kv/          線形化可能KVステートマシン + クライアントセッション管理
├── sim/         決定論的シミュレータ (仮想時間イベントキュー、シード付きPRNG)
│                └── 障害注入: メッセージ喪失/重複/順序入替/遅延、ネットワーク分断、
│                    ノードクラッシュ&再起動、クロックスキュー
├── checker/     不変条件チェッカー + 線形化可能性チェッカー (自作、WGLアルゴリズム)
└── cmd/raftsim  CLI: シード実行、ソークテスト、失敗シードの再現・トレース出力
```

### 3.1 Raftコアのインターフェース (概形)

```go
type Node struct { ... }                     // 1ノード分のRaft状態

type Input interface{}                       // MsgRecv | TimerFire | ClientPropose | ...
type Output struct {
    Messages   []Message                     // 送信すべきメッセージ
    Persist    *HardState                    // fsyncしてから次に進むべき永続状態
    Applied    []Entry                       // ステートマシンにapplyすべきエントリ
    Snapshot   *SnapshotAction               // スナップショット関連の指示
}

func (n *Node) Step(in Input) Output         // 決定論的: 同じ状態+同じ入力 → 同じ出力
```

### 3.2 シミュレータの決定論保証

- 仮想時間: イベントキュー駆動。`time.Now()`・`time.Sleep()` は使用禁止 (CIでgrep検査)
- 乱数: すべて `rand.New(rand.NewSource(seed))` から派生。1シード = 1宇宙
- 並行性: シミュレーションループは単一goroutine。Raftコアにgoroutineがないため競合が存在しない

### 3.3 障害モデル

| 障害 | パラメータ |
|---|---|
| メッセージ喪失 | 確率 p_drop (0〜30%) |
| メッセージ重複 | 確率 p_dup (0〜10%) |
| メッセージ遅延 | 遅延分布 (指数分布、最大数election timeout分) — 順序入替はこれにより自然発生 |
| ネットワーク分断 | ランダムな2分割/3分割を一定仮想時間、その後回復 |
| ノードクラッシュ | 揮発状態を全喪失。fsync済み永続状態のみで再起動 |
| fsync境界クラッシュ | Persist指示の途中(fsync前)でのクラッシュ → 直前のfsync時点まで巻き戻る |
| クロックスキュー | ノードごとのタイマー速度を±20%変動 |

## 4. 検証 (このプロジェクトの本体)

### 4.1 安全性不変条件 (毎イベント後に全ノードの状態を検査)

論文Figure 3の5性質をそのまま機械検査する:

1. **Election Safety**: 同一termに2人のリーダーが存在しない
2. **Leader Append-Only**: リーダーは自分のログを上書き・削除しない
3. **Log Matching**: 2つのログが同index同termのエントリを持つなら、それ以前は完全一致
4. **Leader Completeness**: commitされたエントリは以後の全リーダーのログに存在する
5. **State Machine Safety**: 同indexで異なるコマンドをapplyしたノードが存在しない

加えて: commitIndexの単調性、termの単調性、メンバーシップ変更中の多数決集合の正当性。

### 4.2 線形化可能性チェッカー (自作)

- クライアント操作履歴 (invoke/response の実時間区間 + 結果) を記録
- Wing & Gong アルゴリズム + P-compositionality (キーごとに分割して検査) を自作実装
- 自作チェッカー自体の正当性は、既知の合法/違法履歴のテストベクタで単体検証する

### 4.3 シナリオスイート (ランダムに加え、狙い撃ちの決定的シナリオ)

- 論文 **Figure 8** のシナリオ (旧termエントリの間接コミット問題) の忠実な再現
- 分断されたリーダーがクライアント応答できないこと (stale read防止)
- スナップショット転送中のクラッシュ
- メンバーシップ変更中のリーダークラッシュ、新旧構成の同時多数決
- リーダー自身をクラスタから除去する変更
- 5ノード中2ノード永久停止での可用性継続、3ノード停止での安全な停止

### 4.4 成功基準 (Definition of Done)

- [ ] `make verify` 一発で全検証が走る
- [ ] ランダムソーク: **10,000シード** × 3/5/7ノード混合 × 全障害有効、違反ゼロ
- [ ] 狙い撃ちシナリオスイート全通過
- [ ] 任意の失敗シードについて `raftsim replay -seed N` で決定的に再現し、イベントトレースを出力できる
- [ ] 線形化チェッカーの自己テスト (合法/違法履歴ベクタ) 全通過
- [ ] Raftコアの単体テストカバレッジ 90%以上
- [ ] 外部依存: 標準ライブラリのみ (go.modのrequireが空)

## 5. マイルストーン

| # | 内容 | 受け入れテスト |
|---|---|---|

## 6. 実装規約

- Go 1.22+、`golangci-lint` 相当の標準的なスタイル
- Raftコアパッケージに `time`, `math/rand` (グローバル), `sync` のimportを禁止 (CIで機械検査)
- すべての乱数・時刻はシミュレータから注入
- 各マイルストーン完了時にテストを全実行してから次へ進む
